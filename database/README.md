# TableTop (Smart Shelf) — POS Database

MSSQL schema for the restaurant-side POS, organized around the four app modules.
Matches the conventions the Express repos already use: `NVARCHAR(64)` uuid PKs
(generated by `uuid v4` in `genericRepository.js`), soft delete via `is_deleted`,
`created_at` defaulted by the DB.

## Running

Shortcut: `npm run db:setup` from the repo root does all three steps below
(server/database taken from `.env`, defaulting to LocalDB). Manually:

```powershell
# 0. Create the DB if it doesn't exist yet
sqlcmd -S "(localdb)\MSSQLLocalDB" -Q "IF DB_ID('tabletop') IS NULL CREATE DATABASE tabletop;"

# 1. Schema (DROPS and recreates all POS tables — back up first!)
#    -I is required (filtered indexes need QUOTED_IDENTIFIER ON)
sqlcmd -S "(localdb)\MSSQLLocalDB" -d tabletop -b -I -i database\01_schema.sql

# 2. Seed data (roles, admin user, tables, menu, inventory, sample order)
sqlcmd -S "(localdb)\MSSQLLocalDB" -d tabletop -b -I -i database\02_seed.sql
```

For the shared dev server, swap `-S "(localdb)\MSSQLLocalDB"` for
`-S RIVS-LPLEGION5\MSSQLSERVER2019 -E`. Seed admin login:
`admin@smartshelf.local` / `admin123` (bcrypt-hashed) — change it in production.

## Module → table map

| Module | Tables | Views |
|---|---|---|
| **Customer** (tablet) | `tables`, `categories`, `menu_item`, `orders`, `order_details`, `waiter_calls`, `feedback` | — |
| **Waiter** (dashboard) | `waiter_calls`, `orders`, `order_details`, `staff` | `vw_waiter_board` |
| **Kitchen** (KDS) | `orders`, `order_details`, `order_process` | `vw_kitchen_queue` |
| **Admin** (owner) | `inventory`, `recipes_inventory`, `inventory_logs`, `discounts`, `orders_and_discounts`, `payments`, `users`, `roles`, `staff` | `vw_low_stock`, `vw_daily_sales`, `vw_item_sales` |
| Shared / auth | `users`, `roles`, `roles_and_users`, `staff`, `user_logs` | — |

## How the flow maps to rows

1. **Customer orders** → one `orders` row + one `order_details` row per item
   (`status = 'queued'`). The `notes` column carries special requests.
2. **Kitchen** works the queue (`vw_kitchen_queue`) and updates each
   `order_details.status`: `queued → in_progress → ready_to_serve`.
3. **Waiter** watches `vw_waiter_board` (items ready, pending calls, open
   orders per table), serves, sets items to `served`. "Call waiter" taps land
   in `waiter_calls` (`pending → acknowledged → resolved`).
4. **Payment** → `payments` row (`cash | card | gcash | maya`), order goes
   `paid`. Stock deductions are logged per line item in `inventory_logs`
   (signed `quantity_change`, driven by `recipes_inventory`).
5. **Admin** reads `vw_daily_sales`, `vw_item_sales`, `vw_low_stock`.

## Status values (enforced by CHECK constraints)

| Column | Values |
|---|---|
| `orders.status` | `placed`, `in_progress`, `ready`, `served`, `paid`, `cancelled` |
| `order_details.status` | `queued`, `in_progress`, `ready_to_serve`, `served`, `cancelled` |
| `waiter_calls.status` | `pending`, `acknowledged`, `resolved` |
| `waiter_calls.request_type` | `call_waiter`, `request_bill`, `assistance`, `other` |
| `payments.status` | `pending`, `paid`, `refunded`, `voided` |
| `payments.payment_method` | `cash`, `card`, `gcash`, `maya`, `other` |
| `menu_item.status` | `available`, `unavailable` |

## Deliberate changes vs. the POS.sql draft

- **`roles_and_users` PK renamed** `role_and_users_id` → `roles_and_users_id`
  to match `roleandstaffRepository.js` (the draft had a typo).
- **`order_details.menu_id` → `menu_item_id`** for consistent naming.
- **`orders.payment_method` (smallint) removed** — payments are now their own
  `payments` table with amount/method/status, which sales analytics needs.
- **`staff.role` (smallint) → `staff.role_id` FK** to `roles`; added optional
  `staff.user_id` so waiters/kitchen staff can have login accounts.
- **`inventory` consolidated**: the draft's redundant `in_stock` + `quantity` +
  `unit_price` became `quantity_in_stock`, `reorder_level` (drives
  `vw_low_stock`) and `unit_cost`.
- **`inventory_logs` generalized**: signed `quantity_change` + `change_type`
  (`order_deduction`, `restock`, `adjustment`, `spoilage`) instead of only
  linking order details to recipes.
- **`order_process.duration` (datetime) → `started_at` / `completed_at`** so
  prep times are actually computable.
- **Added**: `waiter_calls` (the "call waiter" feature had no table),
  `order_details.notes` (custom requests), `feedback.rating` (1–5),
  filtered unique indexes on `users.email` / `tables.table_name`.
- **Left out (vendor-platform tables)**: `restaurant`, `admin`, `technician`,
  `ticket`, `licensing`, `device_deployment`, `installation_service`,
  `compliant_resolution`, `logs`, `inventory_device`, and the vendor-level
  `payment` table. Those belong to the SmartShelf company back-office, not the
  in-restaurant POS the four modules describe — say the word if you want a
  separate schema for that side.

## Notes for the API layer

- `genericRepository.update()` does not set `updated_at` — pass it in the
  entity (e.g. `updated_at: new Date()`) when updating.
- `order_details.total` is not computed by the DB — the service layer should
  set `total = price * quantity`.
- Deduct inventory when an order is placed (or fired): for each line item,
  read `recipes_inventory`, subtract `amount_needed * quantity` from
  `inventory.quantity_in_stock`, and write a negative `inventory_logs` row.
